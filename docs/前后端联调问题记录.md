# GoIM 前后端联调问题记录

## 1. 使用目的

本文记录 `dev` 分支前后端联调期间发现的所有问题、处理决定和回归证据。问题修复后不删除记录，而是更新状态、关联提交和回归结果，确保联调结论可以追溯。

联调阶段和检查项以《前后端联调实施计划》为准。

## 2. 处理规则

- 发现问题后先登记编号，再开始修改代码。
- P0、P1 或阻塞当前/后续阶段的问题必须当场修复并回归，通过后才能继续。
- 不阻塞后续验证的 P2、P3 问题可以暂缓，但必须明确处理决定和计划。
- 一个问题对应一个主要根因；不同根因拆分编号。
- 修复后至少执行原复现步骤，并运行受影响模块的自动化测试。
- 状态变更顺序通常为：`待确认 → 待修复 → 修复中 → 待回归 → 已关闭`。
- 不写入访问令牌、刷新令牌、密码或其他敏感信息。

## 3. 严重级别

| 级别 | 定义 | 阶段处理 |
| --- | --- | --- |
| P0 | 服务不可用、严重数据损坏、隐私或安全漏洞 | 立即停止联调，修复并全量回归 |
| P1 | 认证、私聊、群聊等核心业务闭环不可用 | 暂停依赖阶段，修复并回归 |
| P2 | 局部功能异常但存在绕行方式 | 评估依赖关系，可修复后继续或登记暂缓 |
| P3 | 视觉、提示、文档、兼容性或低影响体验问题 | 可集中处理，但最终验收前需给出结论 |

## 4. 状态说明

| 状态 | 含义 |
| --- | --- |
| 待确认 | 已观察到异常，正在收集复现证据 |
| 待修复 | 已确认根因和预期行为，尚未开始修改 |
| 修复中 | 正在修改代码或配置 |
| 待回归 | 修复已完成，等待执行复现步骤和相关测试 |
| 已关闭 | 修复及回归均通过 |
| 暂缓 | 不阻塞当前联调，已记录后续处理决定 |
| 不修复 | 经确认属于预期行为或本轮范围外，并已说明理由 |

### INT-001：前端离线同步未使用复合游标

| 项目 | 内容 |
| --- | --- |
| 阶段 | 0 基线冻结与静态检查 |
| 级别 | P1 |
| 状态 | 已关闭 |
| 首次发现 | 2026-07-11 |
| 影响模块 | WebSocket、断线重连、离线消息分页 |
| 测试账户/资源 | 不适用（静态契约核对发现） |
| 关联提交 | 待联调修复提交 |

**问题描述**

后端在 BUG-013 修复中已将离线同步分页升级为 `syncTime + syncMsgId` 复合游标，但前端的 `SyncReq`、`SyncBatch` 类型和状态仓库仍只有 `syncTime`。自动翻页和重连请求因此没有回传 `lastSyncMsgId`。

**复现步骤**

1. 对照 `backend/internal/model/message.go` 与 `frontend/goim-ws-types.ts`。
2. 检查 `RealtimeBootstrap` 的首次同步和 `hasMore` 自动翻页请求。
3. 检查 `chatStore.applySyncBatch` 保存的游标状态。

**期望结果**

前端保存服务端每批返回的 `syncTime` 和 `syncMsgId`，后续同步请求分别以 `lastSyncTime` 和 `lastSyncMsgId` 原样回传。

**实际结果**

前端只保存和回传 `syncTime`，分页边界或同毫秒多消息场景可能重复或遗漏消息。

**证据**

- 前端类型原先没有 `lastSyncMsgId` / `syncMsgId`。
- 后端 `SyncReq.LastSyncMsgID`、`SyncBatch.SyncMsgID` 已用于严格向后分页。
- `RealtimeBootstrap` 原先只从 store 读取 `lastSyncTime`。

**根因与处理决定**

前端界面分支基于旧版 WebSocket 类型开发，后端分支随后修复了 BUG-013；两条分支合并时没有文本冲突，但产生了契约级语义差异。该问题会阻塞阶段 6，按 P1 当场修复。

**修复内容**

- HTTP 无关；仅更新 WebSocket TypeScript 类型、聊天 store 和自动同步请求。
- store 同时维护 `lastSyncTime` 与 `lastSyncMsgId`。
- 自动翻页和重连均回传完整复合游标。
- 同步更新协议文档并补充 store 测试断言。

**回归结果**

- 原复现步骤：类型、store 和自动翻页请求均已使用完整复合游标。
- 相关自动化测试：2026-07-11，前端 9 项测试、类型检查和生产构建通过。
- 真实同毫秒分页：安排在阶段 6 验证。

### INT-002：开发模式受保护路由自动绕过登录

| 项目 | 内容 |
| --- | --- |
| 阶段 | 0 基线冻结与静态检查 |
| 级别 | P1 |
| 状态 | 已关闭 |
| 首次发现 | 2026-07-11 |
| 影响模块 | 路由认证、真实联调入口 |
| 测试账户/资源 | 未登录用户 |
| 关联提交 | 待联调修复提交 |

**问题描述**

在 Vite 开发模式中，未登录用户直接访问 `/app/...` 时，`ProtectedRoute` 会自动调用 `enterPreview()`，允许进入模拟数据页面。这会绕过真实认证流程，使联调人员可能误把预览数据当成后端数据。

**期望结果**

未登录访问受保护页面始终跳转 `/login`；模拟预览只能由登录页上的显式按钮开启。

**实际结果**

开发模式下自动进入预览，无需用户选择。

**根因与处理决定**

前端开发阶段为方便查看界面加入了自动预览分支，正式联调前未移除。该行为阻塞认证联调，按 P1 当场修复。

**修复内容**

移除 `ProtectedRoute` 的自动 `enterPreview()`；保留登录页显式“进入界面预览”按钮。

**回归结果**

- 自动化测试、类型检查与构建：2026-07-11 全部通过。
- 浏览器未登录路由验证：安排在阶段 1/2。

### INT-003：切换账户时聊天数据未隔离

| 项目 | 内容 |
| --- | --- |
| 阶段 | 0 基线冻结与静态检查 |
| 级别 | P0 |
| 状态 | 已关闭 |
| 首次发现 | 2026-07-11 |
| 影响模块 | 登录、退出、聊天 store、隐私隔离 |
| 测试账户/资源 | 用户 A 切换到用户 B |
| 关联提交 | 待联调修复提交 |

**问题描述**

聊天 store 的 `initializeLive()` 只判断当前是否为 live 模式，不识别所属用户。A 退出或 Token 被替换后登录 B，store 仍可能保留 A 的会话、消息和同步游标，造成跨账户数据展示或同步遗漏。

**期望结果**

聊天状态绑定当前用户 ID；用户变化或退出登录时清空会话、消息、未读数和同步游标。

**实际结果**

live 模式一旦初始化便不再重置，状态没有用户作用域。

**根因与处理决定**

聊天状态设计缺少账户维度，认证清理仅清除 auth store。该问题涉及用户隐私，按 P0 当场修复。

**修复内容**

- chat store 新增 `liveUserId` 和 `resetSession()`。
- `initializeLive(userId)` 在用户变化时重置所有聊天数据与同步游标。
- 无有效会话时由 `RealtimeBootstrap` 断开 WebSocket 并清空聊天状态。
- 新增跨用户切换的 store 单元测试。

**回归结果**

- 自动化测试、类型检查与构建：2026-07-11，新增跨账户状态清理测试，前端 9 项测试、类型检查和生产构建全部通过。
- A 退出后登录 B 的浏览器验证：安排在阶段 2。

### INT-005：缺少 favicon 导致浏览器控制台 404

| 项目 | 内容 |

| 影响模块 | 前端 HTML、浏览器控制台 |

React 开发模式 StrictMode 会同步执行一次 effect 的挂载和清理探测。`RealtimeBootstrap` 在首次 setup 立即连接，probe cleanup 又在握手完成前关闭，浏览器因此警告。该问题不影响最终连接，但干扰联调观测，按 P3 当场修复。

**修复内容**

将真实 WebSocket 连接延迟到当前事件循环结束；StrictMode 的同步 probe cleanup 会先取消定时器，因此只在稳定挂载后建立一次连接。

**回归结果**

- 前端测试、类型检查和构建：2026-07-11 全部通过。
- 浏览器控制台与连接次数：用户强制刷新后确认无新的握手前关闭警告。
