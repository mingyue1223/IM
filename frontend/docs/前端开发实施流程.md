# GoIM 前端开发实施流程（首期）

## 1. 目标与技术基线

目标是在不改变既有后端契约的前提下，构建桌面 Web IM 客户端。优先复用 `frontend/api/`、`goim-api-types.ts` 与 `goim-ws-types.ts`，以 REST 承载资源和配置操作，以 WebSocket 承载实时消息。

建议基线：Vite、React、TypeScript、React Router、Zustand、TanStack Query、Tailwind CSS 和 Framer Motion。具体依赖版本在初始化时固定，并以项目实际脚手架为准。

## 2. 实施原则

- 先用模拟数据完成页面结构、视觉与状态，再接入真实接口。
- 组件按业务域组织，避免把 API、WebSocket 和页面状态混在单个页面组件中。
- REST 请求使用现有 `GoIMApiClient`；访问令牌、刷新和 401 处理集中管理。
- WebSocket 由单一连接管理模块维护，页面组件不直接创建连接。
- 首期只实现文字消息与文字动态；不得以临时 Base64 方式绕开缺失的媒体上传能力。

## 3. 阶段一：设计令牌与应用基础

### 工作内容

1. 初始化 React 工程、路由、基础 CSS 与环境变量。
2. 定义颜色、字体、字号、间距、圆角、阴影、层级、断点和动效时长等设计令牌。
3. 建立按钮、输入框、头像、菜单、抽屉、弹窗、空状态、骨架屏、Toast 等基础组件。
4. 实现浅色主题和“减少动态效果”适配。

### 交付与验收

- 可运行的应用壳和统一基础组件。
- 设计令牌可被全局复用；无深色模式分支。
- 基础交互具有键盘焦点和可访问名称。

## 4. 阶段二：三栏应用壳与模拟界面

### 工作内容

1. 实现聊天、联系人、朋友圈、设置的导航与受保护路由框架。
2. 实现三栏布局、右侧详情抽屉和桌面最小宽度提示。
3. 使用模拟数据完成登录、会话、消息、联系人、群资料、朋友圈和设置的静态/交互视觉。
4. 完成加载、空、错误、无权限等状态的视觉规范。

### 交付与验收

- 不依赖后端即可展示完整首期界面流程。
- 导航、抽屉、弹窗、列表和主要动效符合《前端界面效果说明》。
- 视觉评审通过后再开始逐域接入，避免在接口接入期间反复重做布局。

## 5. 阶段三：认证与 API 基础接入

### 工作内容

1. 接入注册、登录、Token 刷新及退出登录。
2. 集中实现 Token 持久化、401 刷新/失效处理和 API 基址配置。
3. 接入当前用户基础信息、头像 URL 处理与设置读取。
4. 建立 TanStack Query 的查询键、缓存失效和错误反馈约定。

### 交付与验收

- 能完成注册、登录、刷新、退出与登录态恢复。
- 令牌失效能可靠回到登录页，不遗留活动 WebSocket。
- 本地环境遵循《前后端联调指南》的 `18080` 服务地址，避免与后端示例端口混淆。

## 6. 阶段四：私聊文字消息闭环

### 工作内容

1. 实现 WebSocket 连接管理：鉴权连接、连接状态、指数退避重连、被踢出处理和重连后同步。
2. 实现消息状态机：本地 `pending`、`sent`、`failed`；为重试保留同一个客户端消息 ID。
3. 处理 `msg`、`serverAck`、`syncBatch`、`convSync`、`msgRevoked`、`error` 与 `kick` 事件。
4. 完成会话列表、文字消息发送、离线同步、送达确认、已读确认、撤回和失败重试。
5. 完成消息列表滚动、新消息提示与历史阅读不被打断的交互。

### 交付与验收

- 两个账户可稳定收发私聊文字，刷新/断线重连后不重复展示消息。
- 收到确认后待发送消息变为已发送；业务错误时可见失败并能重试。
- 单设备登录踢出后清理状态并停止自动重连。
- 不发送图片、视频、Base64 或其他非文字消息。

## 7. 阶段五：联系人与完整群管理

### 工作内容

1. 接入好友列表、好友申请、接受/拒绝、删除好友、拉黑/解除拉黑。
2. 从联系人详情发起私聊，并与会话状态联动。
3. 接入创建群、群资料、成员列表、添加/移除成员、角色调整和退群。
4. 将群主、管理员、成员权限映射为相应的可见操作与确认流程。

### 交付与验收

- 好友操作成功后列表和详情及时刷新。
- 群资料抽屉按角色正确显示管理能力。
- 所有危险操作均有确认和成功/失败反馈。

## 8. 阶段六：文字朋友圈、设置与上传

### 工作内容

1. 接入文字动态发布、Feed 加载、可见性、点赞、取消点赞、评论、删除评论。
2. 实现分页/游标加载、空状态和错误重试。
3. 接入通知偏好、消息预览、免打扰会话和退出登录。
4. 接入既有头像上传接口，并将相对 URL 与静态资源基址拼接。

### 交付与验收

- 可发布、浏览和互动文字动态，隐私选项可用。
- 头像可上传并正确显示。
- 不实现动态图片上传或聊天媒体上传。

## 9. 阶段七：体验打磨、测试与发布准备

### 工作内容

1. 审查动效节奏、减少动态效果适配、空/错/加载状态和视觉一致性。
2. 验证慢网络、接口错误、WebSocket 中断、Token 过期和重复点击场景。
3. 完成关键组件与状态管理单元测试；对登录、私聊、好友、群管理和朋友圈建立端到端流程测试。
4. 优化长消息列表渲染、缓存策略、代码分割和生产构建产物。
5. 配置生产环境 API/WS/静态资源地址，并验证后端的 SPA 静态托管约定。

### 交付与验收

- 关键首期流程具备自动化覆盖，且能在真实后端环境完成手工验收。
- 不出现阻断性空白页、重复消息、无限重连或无反馈的失败操作。
- 生产构建可被后端 `frontend/dist` 静态托管规则正确加载。

## 10. 后续迭代：媒体消息

媒体消息不属于本流程的任何首期阶段。后续需要前后端共同完成：聊天/动态通用上传端点、格式和大小校验、访问控制、资源生命周期、缩略图或压缩策略、上传进度与取消、`msgType: 2` URL 消息渲染、预览与失败重试。完成契约后再在前端开放图片入口。
