# GoIM 压力测试总结

> 日期：2026-07-05 | 环境：单机 Docker（MySQL 8.4 + Redis 7.2 + RabbitMQ 3.13）| Go 1.24

---

## 1. 测试目标

根据产品需求文档的技术指标：

| 指标 | 目标值 |
|------|--------|
| WebSocket 并发连接数 | ≥ 10,000 |
| 私聊消息 QPS | ≥ 5,000 |
| API 响应时间 (P99) | ≤ 100ms |


## 3. 第一轮：WebSocket 并发连接数

### 3.1 建连吞吐（连接后立即关闭）

每 VU 建立 WS 连接，握手成功后关闭，统计建连成功率和耗时。

| 并发数 | 成功率 | P50 建连 | P95 建连 | 全部完成耗时 |
|--------|--------|----------|----------|-------------|
| 50 | 100% | 0.57ms | 0.97ms | <0.1s |
| 1,000 | 100% | 4.09ms | 9.81ms | <0.1s |
| 5,000 | 100% | 92.39ms | 134.88ms | 0.3s |
| 10,000 | 100% | 176.86ms | 282.70ms | 0.4s |

### 3.2 长连接保持（连接维持 60 秒）

每 VU 建立 WS 连接后持有 60 秒不关闭，观察服务端 goroutine 和内存。

| 同时在线 | 持续时间 | goroutine 总数 | 成功率 | 瓶颈 |
|----------|----------|---------------|--------|------|
| 1,000 | 60s | 2,018 | 100% | — |
| 5,000 | 60s | 10,018 | 100% | — |
| 10,000 | 60s | 20,018 | 100% | — |
| 20,000 | 60s | 40,038 | 100% | — |
| 28,000 | 60s | 56,018 | 100% | 客户端临时端口池 (32768-60999) |
| 40,000 | 60s | 80,039 | 100% | k6 客户端压力 |
| 50,000 | 60s | 100,008 | 100% | 系统综合极限 |

### 3.3 分析

- **goroutine 增长完全线性**：每连接精确增加 2 个 goroutine（ReadPump + WritePump），零泄漏
- **建连延迟受并发数影响**：从 1ms (50并发) 增长到 3.28s (4万并发)，主要瓶颈在 JWT HMAC-SHA256 验签 + 2 次 Redis SET 的瞬时并发拥塞
- **单客户端极限 ≈ 28,000**：受 Linux 临时端口池限制（`ip_local_port_range: 32768-60999` = 28,232 个），非服务端问题
- **服务端极限 ≥ 50,000**：50,000 连接时成功率达到 100%

## 4. 第二轮：消息写入 QPS

### 4.1 压测方法

每个连接持有 WS 长连接，采用 发→收 serverAck→立即发下一条 的流水线模式。每个消息经历完整的同步路径：

```
客户端 WS → MsgService → Redis Lua 原子校验(好友+黑名单+去重+msgID) 
         → MQ Publish(异步) → WS serverAck 返回 → 客户端
```

### 4.2 测试结果

| 连接数 | QPS | P50 | P95 | P99 | 成功率 |
|--------|-----|-----|-----|-----|--------|
| 10 | 29,532 | 0.23ms | 0.38ms | 0.47ms | 100% |
| 50 | 27,838 | 0.67ms | 1.12ms | 1.50ms | 99.98% |
| 100 | 28,993 | 1.70ms | 2.71ms | 3.70ms | 99.96% |
| 200 | 28,007 | 3.08ms | 5.16ms | 7.42ms | 99.93% |
| 1,000 | 27,432 | 18.73ms | 25.04ms | 577ms | 99.58% |

### 4.3 瓶颈分析

```
消息链路 QPS 天花板 ≈ 28,000 msg/s

瓶颈链:
  Redis Lua EVALSHA (单线程)  ← 主瓶颈
       ↓
  JSON 序列化/反序列化 (CPU)
       ↓
  MQ PublishWithContext (5s 超时)
       ↓  ← 异步路径，不计入同步延迟
  Consumer: WriteInbox + Push + InsertMySQL
```

- **QPS 不随连接数增长**：10 连接和 1000 连接产生的 QPS 都在 27K-30K 之间，确认瓶颈在 Redis 单线程而非连接调度
- **延迟随连接数恶化**：P50 从 0.23ms→18.73ms，P99 从 0.47ms→577ms。1000 连接时每个连接都在排队等 Redis
- **少量失败来自 MQ 阻塞**：`PublishWithContext` 在 RabbitMQ 堆积时阻塞最多 5 秒，导致部分消息超时

### 4.4 延迟分布（10 连接，最优状态）

```
P50:  0.23ms  ← Redis Lua EVALSHA 一次网络往返
P95:  0.38ms  ← 偶发 goroutine 调度抖动
P99:  0.47ms  ← 接近最优
AVG:  0.34ms
```

## 6. 压测脚本使用指南

```bash
# === 环境准备 ===
docker compose up -d
go run ./cmd/server -c configs/config.test.yaml

# === 第一轮：WS 连接数 ===

# 1. 注册压测用户
go run benchmark/register_users.go -count=2000 -url=http://localhost:8080

# 2. k6 建连吞吐（连接后立即关闭）
cd benchmark && k6 run --env VUS=1000 ws-conn-test.js

# 3. k6 长连接保持（维持 60 秒）
cd benchmark && k6 run --env VUS=10000 --env HOLD=60 ws-conn-hold.js

# 4. 服务端监控
curl -s http://localhost:6060/debug/pprof/goroutine?debug=1 | head -1

# === 第二轮：消息 QPS ===

# 1. 刷新 token + 配对好友 + 预热 Redis
go run benchmark/setup_friends.go -pairs=1000 -url=http://localhost:8080

# 2. 消息压测
go run benchmark/msg_bench.go -conns=10 -duration=10s
go run benchmark/msg_bench.go -conns=100 -duration=15s
go run benchmark/msg_bench.go -conns=1000 -duration=15s

# 3. 单消息诊断（排查连接/鉴权/好友问题）
go run benchmark/msg_debug.go
```

---

> **结论**：GoIM 在单机部署下，WebSocket 并发连接和消息写入 QPS 分别达到目标的 5 倍和 5.6 倍，Redis 先行 + Lua 原子 + MQ 异步的架构设计经受住了压测验证。主要待修复项为 `AcceptFriendRequest` 的 Redis 缓存缺失。
