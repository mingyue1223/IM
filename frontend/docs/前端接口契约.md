# GoIM 前端接口契约（v1）

本文档以当前后端实现为准，供前端开发与联调使用。Swagger UI（`/swagger/index.html`）提供字段级参考；本文定义跨接口的一致规则和实时通信约定。

## 1. 基础约定

- HTTP 基地址：`/api/v1`；除 `POST /auth/*` 与 `GET /avatar/:userID` 外，均需 `Authorization: Bearer <accessToken>`。
- 请求与响应均为 UTF-8 JSON；上传头像为 `multipart/form-data`，字段名 `file`。
- 成功响应：`{ "code": 0, "message": "ok", "data": T }`；仅提示型成功响应可省略 `data`。
- 失败响应：`{ "code": 非0, "message": "..." }`。前端应优先使用 `code` 判断业务错误，`message` 仅用于展示。
- 列表响应的 `data` 固定为 `{ items: T[], pagination: { total, offset, limit, has_more } }`。
- `created_at`、`updated_at` 为 RFC 3339 UTC 字符串；WebSocket `timestamp`、`lastSyncTime`、`syncTime` 为 Unix 毫秒。

### ID 规则

服务端资源 ID（包括评论 ID）由 MySQL 自增生成，前端以 JSON `number` 接收并原样回传；前端不得自行生成服务端资源 ID。客户端消息去重 ID 使用字符串 `msgId`。

## 3. WebSocket

连接：`GET /ws?token=<accessToken>`。除 `kick` 外，消息统一为 `{ "type": string, "data": object }`。

客户端发送：`msg`、`deliverAck`、`readAck`、`syncReq`、`revokeMsg`、`friendApply`、`ping`。服务端发送：`msg`、`serverAck`、`syncBatch`、`convSync`、`msgRevoked`、`friendAccepted`、`presence`、`error`、`pong`。

关键流程：发送消息后等待 `serverAck`，以 `clientMsgId` 关联临时消息；重连后发送 `syncReq`，分页时同时回传上一批的 `syncTime + syncMsgId` 复合游标，依次处理 `syncBatch` 与 `convSync`；收到 `kick`（当前载荷为 `{ type: "kick", "reason": "new_login" }`）时清除本地登录态。

会话 ID：私聊为 `p_{较小用户ID}_{较大用户ID}`，群聊为 `g_{群组ID}`。完整的 WS 字段类型见 `frontend/goim-ws-types.ts`。

## 4. 前端错误处理

- HTTP 401 或 `code=1003/1106`：清除令牌并跳转登录。
- 修改用户名：`username` 必须为 3～50 个字符且全局唯一；冲突时返回 `409 / code=1103`。修改密码：必须提供当前密码，新密码至少 6 个字符；当前密码错误返回 `401 / code=1105`，前端应保留输入并展示错误。
- `code=4001~4003`：私聊发送失败；`5001~5003`：群聊发送失败，消息应标记为失败且允许重试。
- 请求超时、5xx、WebSocket 断线：保留用户输入或待发消息；重连后通过 `syncReq` 修复状态，不要盲目重复发送已收到 `serverAck` 的消息。
